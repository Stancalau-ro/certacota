## Critical Pitfalls

1. NEVER use BigDecimal.equals() for floor comparisons — use compareTo(). equals() considers scale: BigDecimal("10").equals(BigDecimal("10.00")) is false, but compareTo is 0.
2. NEVER check idempotency key existence before the write (TOCTOU race). Always attempt the INSERT and catch DataIntegrityViolationException. Catch Spring's DataIntegrityViolationException, not JPA's ConstraintViolationException — Spring wraps it.
3. NEVER place @Transactional on the controller — only on AccountServiceImpl. The audit INSERT and balance UPDATE must share one transaction scope.
4. engine-core must have NO Spring annotations. AccountService interface has no @Service. Domain exceptions have no @ResponseStatus. DTOs have no Jackson annotations except where strictly needed for serialisation (Jackson is not on engine-core's classpath anyway — DTOs are plain Java).
5. Use saveAndFlush (not save) for the idempotency key INSERT so that the DB UNIQUE constraint fires before the rest of the method proceeds. If save is used, the constraint violation may surface at transaction commit time, making the catch block unreachable.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | Balance floor enforcement in AccountServiceImpl | mitigate | enforceBalanceFloor uses compareTo (not equals) against effective floor; called before every balance-changing write inside @Transactional; floor check and write are atomic — no TOCTOU |
| T-02-02 | Spoofing | Idempotency key collision (duplicate account creation with same key) | mitigate | UNIQUE constraint on (idempotency_key, operation) is the atomic gate; saveAndFlush triggers the constraint before the business write; DataIntegrityViolationException catch returns the original stored response |
| T-02-03 | Tampering | Audit log integrity (FUND-02) | mitigate | BalanceAuditLog entity: all columns are updatable=false; auditLogRepository.save is inside the same @Transactional scope as accountRepository.save — both commit or both roll back; service layer never calls delete on this repository |
| T-02-04 | Tampering | BigDecimal → Double coercion on JSON input | mitigate | CreateAccountRequest fields use BigDecimal type; Jackson configuration use-big-decimal-for-floats=true in application.yml (Plan 01) prevents Double coercion at deserialisation; floor check uses compareTo not equals |
| T-02-05 | Denial of Service | Arbitrarily large metadata payload in JSONB | accept | No field size limit enforced in Phase 1; Postgres TEXT/JSONB can store multi-MB payloads; caller-trusting API (authentication is platform responsibility per project constraints); can add @Size constraint in Phase 2 if needed |
