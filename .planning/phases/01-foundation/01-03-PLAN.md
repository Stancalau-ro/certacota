As a platform operator, I want to create participant accounts with initial balances, retrieve their committed balances, and close them, so that the token engine can track individual participant positions correctly from day one.

## Critical Pitfalls for This Plan

1. DO NOT put @Transactional on AccountController — it must be on AccountServiceImpl only.
2. DO NOT put @CucumberContextConfiguration or @SpringBootTest on any step definition class — those annotations are exclusively on CucumberSpringConfiguration.
3. In step definitions, use RestTemplate or Spring's TestRestTemplate to call the running service via HTTP (WebEnvironment.RANDOM_PORT starts a real embedded Tomcat). Do NOT use MockMvc.
4. Idempotency-key is a request header (Idempotency-Key: {value}) OR a request body field — per CONTEXT.md specifics, it is a field in the request body (CreateAccountRequest.idempotencyKey). The controller reads it from the @RequestBody, not from a header. Step definitions must set it in the request JSON body.
5. The balance-floor.feature "debit 15.00 from account" step requires a debit endpoint. In Phase 1, there is no /api/v1/accounts/{id}/debit endpoint — discrete transactions are Phase 2. For the floor enforcement test, the step should call account creation with a negative initialBalance (which tests the floor on the initial deposit) OR create a test-only endpoint for the step. Prefer: test floor enforcement by attempting to create an account with initialBalance = -5.00 (below floor 0). If the floor test scenario requires debiting an existing positive balance, implement a minimal internal test hook or defer the debit scenario's floor enforcement step to Phase 2 — annotate the scenario with @Pending in the feature file if the debit endpoint doesn't exist yet. Do NOT implement the full discrete debit endpoint (Phase 2 scope).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | HTTP request body (account ID, balance) | mitigate | @Valid on @RequestBody activates Bean Validation; id and idempotencyKey validated as @NotBlank; initialBalance accepts null (defaults to 0 in service); GlobalExceptionHandler converts MethodArgumentNotValidException to 400 |
| T-03-02 | Tampering | Balance floor bypass via HTTP | mitigate | Floor enforcement is in AccountServiceImpl (not the controller); service is @Transactional — check and write are atomic; no way to bypass floor via HTTP without triggering BalanceFloorViolationException → 422 |
| T-03-03 | Information Disclosure | Error messages in HTTP responses | accept | Error responses return ex.getMessage() which includes the account ID and floor values; the engine trusts caller-supplied IDs (per project constraint: authentication is platform responsibility) — no sensitive PII is in domain exceptions |
| T-03-04 | Denial of Service | Concurrent duplicate account creation with same idempotency key | mitigate | UNIQUE constraint is the gate (Plan 02 ensures this); concurrent duplicate POSTs result in one account created and all duplicates getting the cached response — no duplicate accounts, no error leakage |
| T-03-05 | Tampering | Audit log append-only (FUND-02) | mitigate | Step definition in AuditLogSteps asserts exactly 1 audit entry after creation; auditLogRepository has no delete method called by any service or controller; the Cucumber audit-log.feature is the automated regression guard |
</threat_model>
