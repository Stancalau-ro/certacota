<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP caller → TransactionController | Untrusted amount and metadata values enter here |
| TransactionServiceImpl → AccountRepository | findWithLock issues SELECT FOR UPDATE; must be inside active @Transactional |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | PostTransactionRequest.amount | mitigate | @Positive Bean Validation on amount field added in Plan 02; V4 DDL adds CHECK (amount > 0) as defense-in-depth (this plan) |
| T-02-02 | Tampering | AccountRepository.findWithLock | mitigate | @Lock(PESSIMISTIC_WRITE) serializes all concurrent writers — prevents TOCTOU floor-check bypass; lock is ineffective outside @Transactional (mitigated in Plan 02 by always calling findWithLock inside the @Transactional service method) |
| T-02-03 | Tampering | DiscreteTransaction.metadata (JSONB) | mitigate | @JdbcTypeCode(SqlTypes.JSON) uses Hibernate parameterized binding — no SQL string interpolation; metadata values are stored as-is, never executed |
| T-02-04 | Tampering | Rake arithmetic overflow | mitigate | NUMERIC(38,18) in V4 DDL; Plan 02 applies BigDecimal.setScale(18, RoundingMode.DOWN) on rake computations |
| T-02-05 | Repudiation | idempotency_key on discrete_transactions | accept | idempotency_keys table UNIQUE constraint (V2 migration) catches duplicate writes at DB level; operation-scoped so same key for CREDIT vs DEBIT is permitted by design |
</threat_model>
