<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP caller → TransactionController | Untrusted JSON enters here; @Valid enforces @Positive on amount |
| TransactionServiceImpl → AccountRepository.findWithLock | Row-level lock boundary; all floor checks must happen inside this lock |
| Rake path → to-account / platform-account findById | Credits only — no lock needed; no floor violation possible on credit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | PostTransactionRequest.amount | mitigate | @Positive on amount field + V4 CHECK (amount > 0) — input rejected at controller before service is reached |
| T-02-02 | Tampering | TransactionServiceImpl floor check | mitigate | findWithLock called as first statement in doDebit — floor check AFTER lock acquired; concurrent debits serialize at DB level; TOCTOU window eliminated |
| T-02-03 | Tampering | metadata JSONB values | mitigate | @JdbcTypeCode parameterized binding in DiscreteTransaction entity (Plan 01); metadata values stored but never executed or interpolated into SQL |
| T-02-04 | Tampering | Rake arithmetic overflow | mitigate | BigDecimal.setScale(18, RoundingMode.DOWN) on all rake computations; NUMERIC(38,18) column type rejects values outside precision |
| T-02-06 | Tampering | Rake deadlock (lock ordering) | mitigate | Only from-account uses findWithLock (debit only); to-account and platform-account use plain findById (credits only) — no cycle possible in Phase 2 |
| T-02-07 | Spoofing | Idempotency replay | accept | Operation-scoped idempotency keys prevent double-execution; cross-operation key reuse is permitted by design (different operation strings); same-operation replay returns cached response |
</threat_model>
