    Algorithm (per RESEARCH.md Pattern 4):
    1. Create account "concurrent-debit-001" via POST /api/v1/accounts with initialBalance=200.00 and idempotencyKey="setup-concurrent". If account already exists from a prior run, delete it first via accountRepository.deleteById("concurrent-debit-001").
    2. N = 20, AMOUNT = BigDecimal.TEN (10.00), so 20 × 10 = 200.00 = exact balance
    3. ExecutorService executor = Executors.newFixedThreadPool(N)
    4. CountDownLatch start = new CountDownLatch(1), done = new CountDownLatch(N)
    5. AtomicInteger successCount = new AtomicInteger(0), failCount = new AtomicInteger(0)
    6. For i in 0..N-1: submit task that awaits start, then POSTs DEBIT of AMOUNT with unique idempotency key "concurrent-key-" + i, increments successCount on 201 or failCount otherwise, counts down done
    7. start.countDown() — release all threads simultaneously
    8. done.await(30, TimeUnit.SECONDS)
    9. executor.shutdown()
    10. assertThat(successCount.get() + failCount.get()).isEqualTo(N) — all threads completed
    11. assertThat(successCount.get()).isGreaterThan(0) — at least some succeeded
    12. BigDecimal finalBalance = accountRepository.findById("concurrent-debit-001").get().getBalance()
    13. BigDecimal expectedBalance = new BigDecimal("200.00").subtract(AMOUNT.multiply(BigDecimal.valueOf(successCount.get())))
    14. assertThat(finalBalance.compareTo(expectedBalance)).isEqualTo(0) — no double-spend, no lost update
    15. assertThat(failCount.get()).isGreaterThanOrEqualTo(0) — floor-rejected threads counted (balance might run out before all 20 succeed)

    The test must NOT use Thread.sleep() for synchronization — only CountDownLatch.
    Use java.util.concurrent.ExecutorService, CountDownLatch, AtomicInteger, TimeUnit.
    HTTP calls: POST to http://localhost:{port}/api/v1/transactions with body Map.of("accountId","concurrent-debit-001","type","DEBIT","amount",AMOUNT,"idempotencyKey","concurrent-key-"+i).

    Add to the @BeforeEach: also set restTemplate error handler to suppress non-2xx so that 422 floor-violation responses don't throw exceptions in the thread.
