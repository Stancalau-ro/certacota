<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Application -> Postgres (DDL) | Flyway V10 migration crosses the production schema boundary; an incorrect ALTER could lose JSONB metadata data or leave Hibernate validation broken |
| Caller-supplied JSON metadata -> JPA AttributeConverter | Untrusted JSON payloads cross into MetadataConverter; malformed JSON must fail closed with a deterministic exception (not silent null) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-METADATA-01 | Tampering | V10 ALTER on discrete_transactions.metadata | mitigate | Use `USING metadata::text` cast which Postgres preserves losslessly for JSONB->TEXT (per RESEARCH.md A1). Acceptance criterion: existing discrete-metadata.feature scenarios still pass after migration (proves data preservation through service round-trip). |
| T-4-METADATA-02 | Tampering | MetadataConverter.convertToEntityAttribute | mitigate | Wrap JsonProcessingException as IllegalArgumentException with explicit message, never return partial/null on parse failure. Unit test 7 in Task 1 enforces this. |
| T-4-METADATA-03 | Information Disclosure | MetadataConverter | accept | Converter logs no metadata content; Jackson exception messages may include partial JSON snippets but engine-core has no logging in this class. Risk accepted — no PII contract violation; metadata is caller-supplied opaque payload per META-01. |
| T-4-METADATA-04 | Denial of Service | MetadataConverter (large payload) | accept | No payload-size limit in converter. Spring Boot HTTP request size limit (10MB default) caps inbound metadata; deeper limits not in Phase 4 scope. Risk accepted for Wave 0 — can be tightened in v2 if observed. |
</threat_model>
