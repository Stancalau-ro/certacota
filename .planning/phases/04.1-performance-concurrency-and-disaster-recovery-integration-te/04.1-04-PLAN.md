Docker-java API (transitively on testRuntimeClasspath via Testcontainers 1.21.2):
```
import org.testcontainers.DockerClientFactory;
import com.github.dockerjava.api.DockerClient;
DockerClient client = DockerClientFactory.lazyClient();
client.pauseContainerCmd(containerId).exec();    // SIGSTOP-equivalent
client.unpauseContainerCmd(containerId).exec();  // resume
client.stopContainerCmd(containerId).exec();     // SIGTERM/SIGKILL — container record persists, port mapping preserved (NOT GenericContainer.stop())
client.startContainerCmd(containerId).exec();    // restart same container
```

Per RESEARCH Pitfall 1: NEVER use `redisContainer.stop()` (Testcontainers API) — it deletes the container. Use `DockerClient.stopContainerCmd(id).exec()` so the container record persists with same port binding. CONTEXT.md D-02 was updated to reflect this implementation mechanism while preserving the locked intent ("Full Redis restart while Postgres stays up; engine reconciles StreamRegistry from streaming_transactions WHERE status=ACTIVE on next write").

Engine reconciliation behavior (verified in StreamingServiceImpl + RedisStreamRegistry):
- On Postgres pause: in-flight streams keep running in-memory (StreamRegistry unaffected). Settlement retries succeed when Postgres returns. Engine does NOT auto-terminate per D-03.
- On Redis restart: `StringRedisTemplate` reconnects via Lettuce auto-reconnect. The engine's `onApplicationReady()` reconciliation only fires at Spring context startup — for a runtime Redis restart, the D-13 fallback path (`stopStream` reading rake fields from Postgres) is what makes settlement work. Test must hit `POST /api/v1/streams/{id}/stop` to verify this path.

Endpoints:
- POST `/api/v1/accounts` — setup
- GET `/api/v1/accounts/{id}` — balance + conservation
- POST `/api/v1/streams` — start stream
- POST `/api/v1/streams/{streamId}/stop` — stop with response `StopStreamResponse(settledAmount, stoppedAt, reason)`
