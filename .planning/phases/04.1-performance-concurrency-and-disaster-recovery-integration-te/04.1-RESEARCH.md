## User Constraints (from CONTEXT.md)

### Locked Decisions

**Failure Injection Mechanism**
- D-01: DB connection loss simulated via Testcontainers `PostgreSQLContainer.pause()` / `unpause()`. No extra containers (Toxiproxy not used). No DataSource mock — real TCP-level failure.
- D-02: Two disaster-recovery scenarios: (1) Transient outage — Postgres pause/unpause; (2) Full Redis restart — `GenericContainer.stop()` + `start()` while Postgres stays up. Engine reconciles `StreamRegistry` from `streaming_transactions WHERE status = ACTIVE` on next write.
- D-03: During Postgres pause, in-flight streams keep running in-memory. Settlement retries succeed when Postgres is restored. Test asserts `settled amount = rate × elapsed (not rate × pre-pause time)`. Engine does NOT auto-terminate streams on DB failure.
- D-04: Disaster-recovery tests tagged `@Tag("disaster-recovery")`. Excluded from standard Gradle `test` task. Run explicitly with `./gradlew disasterRecoveryTest` (or equivalent). Configured in `build.gradle`.

## Summary

Phase 04.1 is a greenfield integration test suite layered on top of the existing `engine-service` test infrastructure. No production code changes are required. The test suite adds four concurrency IT classes, one large-database stress IT, and two disaster-recovery IT classes, all driven by a new `application-perf-test.yml` config file. The phase is entirely self-contained within `engine-service/src/test/`.

The key technical challenges are: (1) the disaster-recovery tests require direct docker-java API calls to pause/stop containers because the `GenericContainer.stop()` method deletes the container — the underlying `DockerClientFactory.lazyClient()` API must be used instead; (2) latency tracking uses `HdrHistogram` which is already a transitive dependency via Micrometer; (3) the large-database seeding requires a `ProcessBuilder` invocation of `docker build` at test startup, checking image existence via `docker images -q` first; (4) the token conservation assertion requires iterating all test accounts — no `GET /api/v1/accounts` list endpoint exists, so the test must maintain its own account ID registry.

**Primary recommendation:** Follow the existing `DiscreteTransactionConcurrencyTest` pattern (`CountDownLatch` + `ExecutorService.newFixedThreadPool`) for all new concurrency ITs; use `DockerClientFactory.lazyClient()` for container pause/stop/restart; use `HdrHistogram` via its `Recorder`/`Histogram` API for latency tracking; configure a dedicated `disasterRecoveryTest` Gradle task with `useJUnitPlatform { includeTags 'disaster-recovery' }`.

## Pattern 1: Testcontainers Container Lifecycle — Pause / Unpause (Postgres)

**What:** Simulating a transient DB outage by suspending the Postgres container process using the Docker `pause` API (SIGSTOP equivalent), then resuming it. Container state (data, network membership) is preserved. TCP connections from the JVM to Postgres are blocked during pause.

**API:** `GenericContainer.stop()` permanently removes the container. Do NOT use it for pause simulation. Instead use the underlying `docker-java` client bundled with Testcontainers:

```java
// Source: DockerClientFactory is org.testcontainers.DockerClientFactory (TC 1.21.2)
// [VERIFIED: WebSearch confirmed pattern via Tabnine/Testcontainers community examples]

import org.testcontainers.DockerClientFactory;
import com.github.dockerjava.api.DockerClient;

// Pause — freezes all processes in the container; TCP connections hang
DockerClient client = DockerClientFactory.lazyClient();
client.pauseContainerCmd(postgresContainer.getContainerId()).exec();

// ... test: settle streams, observe retry behavior ...

// Unpause — resumes all processes; existing TCP connections reconnect
client.unpauseContainerCmd(postgresContainer.getContainerId()).exec();
```

**Caveats:**
- `DockerClientFactory.lazyClient()` is preferred over `DockerClientFactory.instance().client()` to avoid triggering a new Docker connection if one isn't already open. [ASSUMED — based on TC community patterns; both forms exist]
- After `unpauseContainerCmd`, the JVM's HikariCP connection pool will reconnect on the next checkout. There is a brief window (typically < 1s for local Docker) before connections are usable. Tests should include a short `Thread.sleep(500)` or a retry loop after unpause before asserting settlement. [ASSUMED — based on connection pool behavior knowledge]
- `getContainerId()` on `PostgreSQLContainer` returns the running container's Docker ID — this is stable for the container's lifetime as long as the container was started with the Testcontainers `@ServiceConnection` or static `@Bean` pattern. [VERIFIED: TC source]
- The `PostgreSQLContainer` in `TestcontainersConfiguration` is declared as an instance method `@Bean`, which means it gets a new container per test class context load. For DR tests that need to call `pause()`/`unpause()`, the container reference must be accessible as a static field or injected via `@Autowired`. The cleanest approach is to expose it as a `static final` field in a shared `ContainerHolder` class (see Pattern 2).

## Pattern 2: Testcontainers Container Lifecycle — Stop / Start (Redis, full restart)

**What:** Simulating a Redis crash and restart by stopping the container (without deletion) and restarting it using the docker-java client directly. This clears Redis in-memory state, triggering the engine's `onApplicationReady()` reconciliation path.

**Key difference from Postgres pause:** `GenericContainer.stop()` on the Testcontainers object REMOVES the container. Use `docker-java` stop/start instead:

```java
// Source: [CITED: https://sirdir.medium.com/testcontainers-restart-of-container-without-deleting-it-f2dd0645e984]
// [VERIFIED: Multiple WebSearch sources confirm DockerClientFactory.lazyClient() pattern]

DockerClient client = DockerClientFactory.lazyClient();

// Stop the container (process-level SIGTERM/SIGKILL — container process ends but container record persists)
client.stopContainerCmd(redisContainer.getContainerId()).exec();

// ... verify engine behavior during Redis outage (expected: 503 on stream operations) ...

// Start the container again — Redis starts fresh (no AOF/RDB configured, so state is cleared)
client.startContainerCmd(redisContainer.getContainerId()).exec();

// Allow Redis to start up before proceeding
Thread.sleep(1000); // or use a wait strategy
```

**Caveats after Redis restart:**
- The `startedAtNanoFromCurrentJvm` flag in `StreamState` will be `false` after reconciliation from DB (because `StreamState.fromDbRow()` sets nano=0). The elapsed time calculation falls back to wall-clock millis (`System.currentTimeMillis() - startedAtEpochMillis`). Test assertions should account for this — the settled amount after restart-reconciliation uses millis precision, not nano precision.
- The `spring.data.redis.*` system properties were set at JVM startup. The restarted Redis container binds to the same host port (Docker container restart preserves port mapping). No config reload needed. [ASSUMED — Docker port binding behavior on restart; verified via community patterns]
- After `startContainerCmd`, `StringRedisTemplate` will reconnect automatically via Lettuce's auto-reconnect. The engine will not call `onApplicationReady()` again — reconciliation is only triggered at Spring context startup. To trigger reconciliation, the test must make a write request that hits `streamRegistry.get()` and follows the D-13 fallback path from `StreamingServiceImpl.stopStream()` (reads rake fields from DB when Redis state missing). [VERIFIED: reading StreamingServiceImpl.stopStream() lines 215-220]

## Pattern 3: Container Reference Exposure for DR Tests

**What:** DR tests need direct access to the `PostgreSQLContainer` and `GenericContainer` (Redis) instances. The existing `TestcontainersConfiguration` keeps `redis` as `static final` but `postgresContainer()` is an instance `@Bean` — the returned container is not stored as an accessible static.

**Recommended approach:** Extend `TestcontainersConfiguration` to expose both container references as package-visible statics, or create a separate `DrTestcontainersConfiguration` for DR tests:

```java
// In TestcontainersConfiguration.java — make postgres a static too
static final PostgreSQLContainer<?> postgres;
static final GenericContainer<?> redis;

static {
    redis = new GenericContainer<>(DockerImageName.parse("redis:7-alpine"))
        .withExposedPorts(6379);
    redis.start();
    // ...

    postgres = new PostgreSQLContainer<>(DockerImageName.parse("postgres:17-alpine"));
    postgres.start();
    // Set datasource properties manually (since @ServiceConnection requires @Bean)
}
```

Alternative: DR test classes can use `@Autowired ApplicationContext` to look up the container bean. The cleanest pattern for this codebase is a dedicated `DrContainerHolder` class with `static` fields, since DR tests need container references before Spring context starts. [ASSUMED — design choice; either approach is valid]

**Important:** Exposing both containers as static fields breaks the `@ServiceConnection` annotation pattern (which requires the `@Bean` return value to be the container). The workaround is to use `@DynamicPropertySource` for both datasource and Redis when containers are managed as static fields. [VERIFIED: existing `TestcontainersConfiguration.java` already uses this pattern for Redis]

## Pattern 5: HdrHistogram for Latency Percentiles

**What:** `HdrHistogram` 2.2.2 is already on the test classpath as a transitive dependency of `io.micrometer:micrometer-registry-prometheus`. No new dependency declaration needed.

```java
// Source: [VERIFIED: ./gradlew :engine-service:dependencies output shows org.hdrhistogram:HdrHistogram:2.2.2]
// HdrHistogram API is stable since 2.x — [ASSUMED: API unchanged between 2.2.2 and current docs]

import org.HdrHistogram.Histogram;

public class LatencyRecorder {

    // Track up to 60 seconds, with 1ms resolution
    private final Histogram histogram = new Histogram(60_000L, 3);

    public void record(long elapsedMs) {
        histogram.recordValue(elapsedMs);
    }

    public long p50() { return histogram.getValueAtPercentile(50.0); }
    public long p95() { return histogram.getValueAtPercentile(95.0); }
    public long p99() { return histogram.getValueAtPercentile(99.0); }
    public long max() { return histogram.getMaxValue(); }
}
```

**Thread safety note:** `Histogram` is NOT thread-safe for concurrent `recordValue()` calls. Use `Recorder` with `flipToSnapshot()` for multi-threaded recording, OR synchronize on the histogram, OR use one histogram per thread and merge at the end. Recommended for this phase:

```java
// Per-thread histograms merged at collection time
// Source: [ASSUMED — standard HdrHistogram concurrent usage pattern]
ConcurrentLinkedQueue<Long> latenciesMsQueue = new ConcurrentLinkedQueue<>();
// Each thread: latenciesMsQueue.add(elapsedMs)
// After all threads finish: sort and compute percentiles manually from the list
// OR: create one Histogram per thread, merge at end:
List<Histogram> perThreadHistograms = new CopyOnWriteArrayList<>();
```

Alternative (simplest, given test context where all threads finish before report): collect all latencies into a `ConcurrentLinkedQueue<Long>`, then at `@AfterAll` sort the list and compute percentiles from index `(int)(n * 0.5)`, `(int)(n * 0.95)`, `(int)(n * 0.99)`. This avoids the thread-safety concern entirely and works correctly for the report. [ASSUMED — implementation choice]

## Pattern 6: Docker Image Check and Build from Java

**What:** At test startup (`@BeforeAll`), check if the seeded Postgres image exists locally, and build it if not.

```java
// Source: [VERIFIED: D-16 decision + standard ProcessBuilder pattern]
// [ASSUMED: ProcessBuilder approach — no standard Testcontainers API for this]

private static void ensureSeedImageExists(String imageName, int accounts, int txnPerAccount)
        throws Exception {

    // Check image existence
    ProcessBuilder check = new ProcessBuilder(
        "docker", "images", "-q", imageName);
    check.environment().put("DOCKER_HOST", "tcp://localhost:2375"); // match test task env
    Process checkProcess = check.start();
    String imageId = new String(checkProcess.getInputStream().readAllBytes()).trim();
    checkProcess.waitFor();

    if (!imageId.isEmpty()) {
        return; // image exists
    }

    // Build if missing
    Path dockerDir = Paths.get("src/test/docker/perf-db");
    ProcessBuilder build = new ProcessBuilder(
        "docker", "build",
        "--build-arg", "ACCOUNTS=" + accounts,
        "--build-arg", "TXN_PER_ACCOUNT=" + txnPerAccount,
        "-t", imageName,
        dockerDir.toAbsolutePath().toString()
    );
    build.redirectErrorStream(true);
    build.inheritIO();
    Process buildProcess = build.start();
    int exitCode = buildProcess.waitFor();
    if (exitCode != 0) {
        throw new IllegalStateException("docker build failed for image: " + imageName);
    }
}
```

**Then use the built image with Testcontainers:**
```java
// For the LargeDatabaseStressIT, override the postgres container to use the seeded image:
static final PostgreSQLContainer<?> seededPostgres =
    new PostgreSQLContainer<>(DockerImageName.parse(imageName)
        .asCompatibleSubstituteFor("postgres"))
    .withEnv("POSTGRES_DB", "certacota")
    .withEnv("POSTGRES_USER", "certacota")
    .withEnv("POSTGRES_PASSWORD", "certacota");
```

**Caveat:** The `asCompatibleSubstituteFor("postgres")` call tells Testcontainers that the custom image should be treated as a Postgres-compatible image, enabling `@ServiceConnection` or `@DynamicPropertySource` connection setup. [VERIFIED: Testcontainers docs on custom images]

## Pattern 8: JSON Report Writing (Jackson)

**What:** Write a structured JSON report at `@AfterAll` using Jackson `ObjectMapper`. Jackson 2.19.1 is already on testRuntimeClasspath.

```java
// Source: [VERIFIED: jackson-databind 2.19.1 on testRuntimeClasspath]

// BigDecimal serialization with 18 decimal places:
// Jackson serializes BigDecimal using toString() by default, which preserves scale.
// A BigDecimal with scale=18 (e.g., "924872.341000000000000000") will serialize correctly.
// No special configuration needed — as long as the BigDecimal was created with scale=18.

ObjectMapper mapper = new ObjectMapper()
    .registerModule(new JavaTimeModule())
    .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);

// Write report:
Path reportDir = Paths.get("target/perf-reports");
Files.createDirectories(reportDir);
String timestamp = DateTimeFormatter.ofPattern("yyyyMMdd-HHmmss")
    .format(LocalDateTime.now());
Path reportFile = reportDir.resolve(scenario + "-" + timestamp + ".json");
mapper.writerWithDefaultPrettyPrinter().writeValue(reportFile.toFile(), reportData);
```

**Note on BigDecimal 18dp:** When summing `BigDecimal` values from `AccountResponse.balance()` (which has `precision=38, scale=18` in the DB), the resulting `BigDecimal` preserves scale through add/subtract operations. The `tokenConservation.delta` check uses `compareTo(BigDecimal.ZERO)` — not `.equals()` — to handle any trailing zeros correctly. [VERIFIED: reading Account.java column definition `precision = 38, scale = 18`]

## Pattern 9: Token Conservation Assertion

**What:** After each test scenario, sum all account balances and verify the conservation invariant. No `GET /api/v1/accounts` (list all) endpoint exists.

**Available endpoint:** `GET /api/v1/accounts/{accountId}` returns `AccountResponse` with `balance` field. [VERIFIED: reading AccountController.java and AccountResponse.java]

**Strategy:** Each IT class maintains a `Set<String>` of account IDs created during `@BeforeAll`. At `@AfterAll`, iterate the set and call `GET /api/v1/accounts/{id}` for each, summing the `balance` field from the JSON response:

```java
// Source: [VERIFIED: AccountController.java @GetMapping("/{accountId}"), AccountResponse.java]

BigDecimal totalFinalBalance = BigDecimal.ZERO;
for (String accountId : testAccountIds) {
    ResponseEntity<AccountResponse> resp = restTemplate.getForEntity(
        "http://localhost:" + port + "/api/v1/accounts/" + accountId,
        AccountResponse.class);
    totalFinalBalance = totalFinalBalance.add(resp.getBody().balance());
}

// Conservation: initialTokensTotal + totalCredited = finalTokensTotal + totalSettled + totalDebited
BigDecimal delta = initialTokensTotal.add(totalCredited)
    .subtract(totalFinalBalance)
    .subtract(totalSettled)
    .subtract(totalDebited);
boolean conserved = delta.compareTo(BigDecimal.ZERO) == 0;
```

**Note on streaming settlement tracking:** Settled amounts come from `StopStreamResponse.settledAmount()`. The test must accumulate settled amounts across all stop calls during the test run. [VERIFIED: reading StopStreamResponse DTO pattern from StreamingServiceImpl]

-- Uses \set to pick up ARG-injected environment at psql init time
-- Source: [ASSUMED — standard Postgres generate_series pattern for test seeding]

-- Accounts: IDs 'stress-acc-000001' through 'stress-acc-{ACCOUNTS}'

    'stress-acc-' || LPAD(i::text, 6, '0'),

## Common Pitfalls

### Pitfall 1: `GenericContainer.stop()` Deletes the Container
**What goes wrong:** Calling `redisContainer.stop()` (Testcontainers API) removes the Docker container. The subsequent `redisContainer.start()` creates a NEW container with a new port mapping. The JVM's Redis connection is now pointing at the wrong port.
**Why it happens:** Testcontainers `stop()` is designed for test teardown — it calls `docker rm` after stopping.
**How to avoid:** Use `DockerClientFactory.lazyClient().stopContainerCmd(containerId).exec()` followed by `startContainerCmd(containerId).exec()`. The container record persists, same port mapping, same state.
**Warning signs:** `spring.data.redis.port` in the application context no longer matches the container's actual mapped port after "restart".

### Pitfall 2: Container ID Retrieved Before Container Starts
**What goes wrong:** `container.getContainerId()` returns `null` if called before the container has started.
**Why it happens:** For static containers, `getContainerId()` is valid after the static block completes (start is called there). For `@Bean` containers, it's valid after the Spring context is ready.
**How to avoid:** Only call `getContainerId()` inside test methods or `@BeforeAll` — never in field initializers or before the context starts.

### Pitfall 3: HdrHistogram recordValue > maxValue Throws
**What goes wrong:** `Histogram.recordValue(value)` throws `ArrayIndexOutOfBoundsException` if `value > maxValue` (the first constructor arg).
**Why it happens:** If a test request takes longer than the configured max (e.g., 60_000ms), the recording fails.
**How to avoid:** Either set a generous max (e.g., `300_000L` for 5 minutes), or catch the exception and record `maxValue` instead, or use `recordValue(Math.min(value, maxValue))`.

### Pitfall 4: BigDecimal Scale Loss in Conservation Assertion
**What goes wrong:** `BigDecimal.ZERO.compareTo(delta)` passes even when delta is `0.000000000000000001` if `delta` was calculated with insufficient scale.
**Why it happens:** `BigDecimal` arithmetic preserves scale through add/subtract but if an intermediate result is created from a `double` or JSON deserialization uses `BigDecimal.valueOf(double)`, scale is lost.
**How to avoid:** Parse JSON response `balance` field as `BigDecimal` using Jackson's `DeserializationFeature.USE_BIG_DECIMAL_FOR_FLOATS` (already configured in `application.yml`). Always sum with `BigDecimal.add()`, never through `double`.

### Pitfall 5: Multiple Spring Contexts for Concurrency and DR Tests
**What goes wrong:** DR tests that manipulate static container state (pause, stop) can corrupt the shared Spring context used by concurrency tests if both suites run in the same Gradle task.
**Why it happens:** Testcontainers static containers are shared across all tests in the same JVM. Pausing the Postgres container mid-run breaks any test that also needs Postgres.
**How to avoid:** Keep DR tests in a separate Gradle task (`disasterRecoveryTest`) that runs in isolation. The `test` and `performanceTest` tasks never include `@Tag("disaster-recovery")` tests.

### Pitfall 6: Docker Build Args Not Reaching SQL Scripts
**What goes wrong:** Postgres `docker-entrypoint-initdb.d/` scripts run as `psql`, which doesn't inherit Docker build args.
**Why it happens:** `ARG` in Dockerfile is a build-time variable. By init time, the container is running — build args are gone unless explicitly stored as `ENV`.
**How to avoid:** Use `ENV` to copy the build args, then use a shell init script (not pure SQL) that uses `$ACCOUNTS` and `$TXN_PER_ACCOUNT` to generate the SQL before running it.

### Pitfall 7: @ActiveProfiles on Multiple Nested Annotations
**What goes wrong:** If a `@SpringBootTest` base class sets `@ActiveProfiles("test")` and a subclass adds `@ActiveProfiles("perf-test")`, only the subclass annotation is used — profiles don't merge across class hierarchy.
**Why it happens:** `@ActiveProfiles` on a subclass overrides the inherited value, it does not append.
**How to avoid:** Declare both profiles in the concrete test class: `@ActiveProfiles({"test", "perf-test"})`. Do not rely on inheritance for profile accumulation.

## Open Questions

1. **Single TestcontainersConfiguration vs. separate DR configuration**
   - What we know: The existing `TestcontainersConfiguration` has Redis as static, Postgres as `@Bean`. DR tests need Postgres as a static too.
   - What's unclear: Changing `TestcontainersConfiguration` to use a static Postgres breaks `@ServiceConnection`. The planner must decide: modify the existing class or create a `DrTestContainerConfig` that both DR test classes import.
   - Recommendation: Create a separate `DrContainerHolder` with two static containers and `@DynamicPropertySource` wiring. Leave `TestcontainersConfiguration` unchanged so existing tests are unaffected.

2. **Virtual threads for MixedConcurrencyIT**
   - What we know: Java 21 is the target JVM. Virtual threads (`Executors.newVirtualThreadPerTaskExecutor()`) are available.
   - What's unclear: Per-account goroutine model in D-11 suggests one thread per account lifecycle. With `N` accounts and multiplier for hot accounts, thread count could reach 50-200. Platform threads handle this easily; virtual threads would be marginally more efficient but add no correctness value.
   - Recommendation: Use platform threads (`Executors.newFixedThreadPool`) for consistency with existing tests. Defer virtual threads per the deferred items list.

3. **Token conservation during streaming: capturing settled amounts**
   - What we know: `StopStreamResponse.settledAmount()` returns the settled amount. But auto-terminated streams settle outside test control.
   - What's unclear: If an auto-termination fires during the test (balance exhaustion), the test thread doesn't see the `StopStreamResponse` — the settlement happens asynchronously via `AutoTerminationScheduler`.
   - Recommendation: Set account initial balances high enough that auto-termination cannot fire during the test window. Or: query the `balance_audit_log` via a dedicated admin endpoint (none exists) — or sum from individual account balance deltas (initial minus final) instead of tracking settlements explicitly.
