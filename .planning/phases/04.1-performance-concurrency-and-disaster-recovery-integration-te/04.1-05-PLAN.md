<objective>
Build the large-database stress integration test (`LargeDbStressIT`) in Wave 3.

This plan was split out of Plan 04 per checker feedback W3 + B5: the original combined plan had each task implementing ~275 LOC of complex integration test code (excess context cost), AND it silently reduced D-15 to a build-pipeline smoke test by running the engine against `postgres:17-alpine` with runtime JDBC seeding instead of the locally-built seeded image. Purpose: Validate SC-3 (large-database suite validates query/write performance against seeded dataset of configurable size).

Output:
- `LargeDbStressIT.java` — auto-built seeded Postgres image, query latency assertions, tagged `performance`

Docker seeded image (Plan 01 — `engine-service/src/test/docker/perf-db/Dockerfile` + `seed.sh`):
- Dockerfile accepts `ARG ACCOUNTS` and `ARG TXN_PER_ACCOUNT` and propagates them as ENV vars
- `seed.sh` (Plan 01, REAL implementation — NOT a scaffold) creates `stress_accounts` (id VARCHAR(36) PK, balance NUMERIC(36,18), created_at TIMESTAMPTZ) and `stress_audit_log` tables, then performs the bulk inserts via `generate_series(1, ${ACCOUNTS})` and `generate_series(1, ${ACCOUNTS} * ${TXN_PER_ACCOUNT})` at container init time (D-17). - The seed runs in `/docker-entrypoint-initdb.d/01-seed.sh` BEFORE any application connects, so by the time `LargeDbStressIT` boots the engine against the container, both tables are fully populated. There is no separate Flyway run against these tables — `stress_accounts`/`stress_audit_log` are stress-test-only tables and are intentionally distinct from the engine's Flyway-managed schema.

Redis wiring: this test does NOT need Redis container manipulation. The engine still needs a Redis instance for streaming state. Two options:
- Option A (recommended): `@Import(TestcontainersConfiguration.class)` for the Redis static + override the Postgres datasource via `@DynamicPropertySource` declared inside `LargeDbStressIT` (Spring concatenates property sources — the test's `@DynamicPropertySource` wins for the keys it declares). The seeded Postgres replaces the `TestcontainersConfiguration` Postgres `@Bean`.
- Option B: declare a dedicated `@TestConfiguration` class (similar to `DrContainerHolder`) with the seeded Postgres + standard Redis; import that instead.

Choose Option A unless a Spring property-source conflict surfaces at runtime — A keeps the change set minimal and reuses Plan 01's existing Redis container.

Endpoints:
- GET `/api/v1/accounts/{id}` — for read-latency load
- POST `/api/v1/accounts/{id}/debit` — for write-latency load (uses smallest debit amount so floor is never hit on a 1,000,000-balance seeded account)

Conservation strategy: with 10,000+ seeded accounts, summing balances via N REST GETs is N requests. Use the seeded container's JDBC datasource for a single `SELECT SUM(balance) FROM stress_accounts WHERE id LIKE 'stress-acc-%'` to compute the conservation total. The test injects the engine's `DataSource` via `@Autowired javax.sql.DataSource` (exception to D-13 — this is setup/verification only, NOT engine business logic). The REST API does not expose a bulk-aggregate endpoint, so JDBC is the only practical option for tens of thousands of accounts. Document this exception in the threat model.

    Static seeded-image container:
    ```
    private static final int ACCOUNTS = Integer.parseInt(System.getProperty("certacota.perf.stress.accounts", "10000"));
    private static final int TXN_PER_ACCOUNT = Integer.parseInt(System.getProperty("certacota.perf.stress.transactions-per-account", "500000"));
    private static final String IMAGE_NAME = "certacota-perf-db-a" + ACCOUNTS + "-t" + TXN_PER_ACCOUNT;

    private static PostgreSQLContainer<?> seededPostgres;

    static {
        try {
            ensureSeedImageExists(IMAGE_NAME, ACCOUNTS, TXN_PER_ACCOUNT);
            DockerImageName seededImage = DockerImageName.parse(IMAGE_NAME)
                .asCompatibleSubstituteFor("postgres");
            seededPostgres = new PostgreSQLContainer<>(seededImage)
                .withDatabaseName("certacota")
                .withUsername("certacota")
                .withPassword("certacota");
            seededPostgres.start();
        } catch (Exception e) {
            throw new IllegalStateException("Failed to provision seeded postgres image " + IMAGE_NAME, e);
        }
    }

    @DynamicPropertySource
    static void registerSeededPostgres(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", seededPostgres::getJdbcUrl);
        registry.add("spring.datasource.username", seededPostgres::getUsername);
        registry.add("spring.datasource.password", seededPostgres::getPassword);
    }
    ```

    Note on static init ordering: declaring `ACCOUNTS`/`TXN_PER_ACCOUNT` via `System.getProperty` with defaults lets CI override via `-Dcertacota.perf.stress.accounts=500`. The values must be available BEFORE `@DynamicPropertySource` fires (Spring resolves dynamic sources during context startup, AFTER static init). Reading them in the static block guarantees image-name correctness.

    Reasoning: static-block override of the datasource means the Spring property `spring.datasource.url` is supplied by `seededPostgres::getJdbcUrl` (an instance from this test's static block), NOT by `TestcontainersConfiguration.postgresContainer()` (the `@Bean` Postgres). Spring's property precedence ensures the `@DynamicPropertySource` value wins. If a Spring conflict surfaces (e.g., `@ServiceConnection` taking precedence over `@DynamicPropertySource`), fall back to Option B: declare a dedicated `@TestConfiguration` class similar to `DrContainerHolder` and `@Import(SeededStressContainerHolder.class)` instead of `TestcontainersConfiguration`.

    `@Test void queryLatencyUnderLargeDataset() throws Exception`:
    1. Determine `int accountsToProbe = Math.min(100, ACCOUNTS);` and `int writesToIssue = Math.min(50, ACCOUNTS);`
    2. Read load: for `i` in `[1, accountsToProbe]`:
       - Compose accountId `"stress-acc-" + String.format("%06d", i)` (matches Plan 01's `seed.sh` LPAD pattern `'stress-acc-' || LPAD(n::text, 6, '0')`)
       - `long t0 = System.currentTimeMillis();`
       - `ResponseEntity<AccountResponse> resp = restTemplate.getForEntity("http://localhost:" + port + "/api/v1/accounts/" + accountId, AccountResponse.class);`
       - `long elapsed = System.currentTimeMillis() - t0;`
       - `readLatency.record(elapsed);`
       - `if (resp.getStatusCode().value() == 200) { successCount.incrementAndGet(); } else { rejectedCount.incrementAndGet(); }`

       Note: stress_accounts is a stress-test-only table; the engine's REST `/accounts/{id}` endpoint queries the engine's primary accounts table (Flyway-managed). The read-latency probe measures Postgres query latency under a large physical row count in the same Postgres instance (stress_audit_log contributes to buffer/cache pressure). If the engine's `accounts` table does not contain `stress-acc-*` IDs, expect `rejectedCount > 0` — that is fine for the latency-under-load measurement, since latency is recorded on every call regardless of status. The conservation invariant uses the stress_accounts JDBC SUM, not the REST 404 responses.
    3. Write load: for `i` in `[1, writesToIssue]`:
       - accountId `"stress-acc-" + String.format("%06d", i)`
       - body `{"amount": new BigDecimal("1"), "idempotencyKey": "stress-debit-" + i}`
       - POST `/api/v1/accounts/{id}/debit`, time the call, `writeLatency.record(...)`
       - On 201: `successCount.incrementAndGet();` AND `totalDebited.updateAndGet(c -> c.add(BigDecimal.ONE));`
       - Else: `rejectedCount.incrementAndGet();`
    4. Assert latency:
       - `long readP95 = readLatency.snapshot().p95Ms();`
       - `long writeP95 = writeLatency.snapshot().p95Ms();`
       - `long ceiling = config.getStress().getMaxQueryLatencyMs();`
       - `assertThat(readP95).as("Read p95 under stress").isLessThanOrEqualTo(ceiling);`
       - `assertThat(writeP95).as("Write p95 under stress").isLessThanOrEqualTo(ceiling * 2);` (writes get 2× ceiling — synchronous floor check + Postgres write is heavier than a read)

    `@AfterAll void teardown() throws Exception`:
    1. Compute initial token total via JDBC against the seeded `stress_accounts` table (Plan 01's `seed.sh` populated balances at scale=18 with the constant `1000000.000000000000000000`):
       ```
       BigDecimal initialTokensTotal;
       BigDecimal totalFinalBalance;
       try (Connection conn = dataSource.getConnection();
            PreparedStatement ps = conn.prepareStatement("SELECT COUNT(*) AS cnt, SUM(balance) AS sum_balance FROM stress_accounts WHERE id LIKE 'stress-acc-%'");
            ResultSet rs = ps.executeQuery()) {
           rs.next();
           int seededCount = rs.getInt("cnt");
           totalFinalBalance = rs.getBigDecimal("sum_balance");
           initialTokensTotal = new BigDecimal("1000000").multiply(BigDecimal.valueOf(seededCount));
       }
       ```
       (Initial balance per account is the constant `1000000.000000000000000000` defined by Plan 01's seed.sh — every seeded row in `stress_accounts` is initialised to 1,000,000 at scale 18. Multiplying by the row count gives the initial pool. Final pool = current SUM. The debit REST calls target engine-managed accounts (not `stress_accounts`), so for the conservation invariant on `stress_accounts` the only balance-changing path is none — `totalDebited` is reported separately for transparency but does not affect the `stress_accounts` SUM.)
    2. `BigDecimal delta = initialTokensTotal.subtract(totalFinalBalance);`
       `boolean conserved = delta.compareTo(BigDecimal.ZERO) == 0;`
    3. Build report `Map<String,Object>`:
       - `scenario="large-db-stress"`, `timestamp=OffsetDateTime.now()`
       - `scenarioConfig=Map.of("accounts", ACCOUNTS, "transactionsPerAccount", TXN_PER_ACCOUNT, "maxQueryLatencyMs", config.getStress().getMaxQueryLatencyMs(), "imageName", IMAGE_NAME)`
       - `readLatency=readLatency.snapshot()`, `writeLatency=writeLatency.snapshot()`
       - `operationCounts=Map.of("success", successCount.get(), "rejected", rejectedCount.get(), "error", 0)`
       - `correctnessVerdicts=Map.of("latencyUnderCeiling", <bool>, "allBalancesCorrect", conserved)`
       - `tokenConservation=Map.of("initialTokensTotal", initialTokensTotal, "finalTokensTotal", totalFinalBalance, "totalSettled", BigDecimal.ZERO, "totalCredited", BigDecimal.ZERO, "totalDebited", totalDebited.get(), "delta", delta, "conserved", conserved)`
    4. `PerfReportWriter.write("large-db-stress", report);`
    5. Assert `conserved` true.
    6. Defensive stop: `seededPostgres.stop();` is NOT called — Testcontainers' Ryuk reaper cleans up the static container on JVM exit. Calling `.stop()` here would delete the container; the next test run's image cache hit (D-16) wouldn't matter because we'd need to rebuild the container anyway. Leave it for Ryuk.

    NEVER bulk-insert from this test class via JDBC — bulk seeding lives in `seed.sh` inside the built image (D-15 + D-17). The test only reads + measures + asserts.
