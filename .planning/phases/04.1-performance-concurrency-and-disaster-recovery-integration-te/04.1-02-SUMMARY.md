# Phase 04.1 Plan 02: Concurrency Integration Tests Summary

Three concurrency-suite integration tests consuming Wave 0 infrastructure. No production code modified; all tests exercise public REST endpoints only.

## What Was Built

### ConcurrentSessionsIT

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/ConcurrentSessionsIT.java`

**Scenario:** N=10 parallel streaming sessions (from `config.getConcurrency().getSessions().getCount()`) each with a randomised duration sampled from `[minDurationMs, maxDurationMs]` (default 1000-5000ms). Each session POSTs `/api/v1/streams`, sleeps for its random duration, then POSTs `/api/v1/streams/{id}/stop`. All N sessions start simultaneously via `CountDownLatch`.

**Conservation check:** Simple — streaming only moves tokens from account to settled; no credits or rake.
`delta = initialTokensTotal - finalTokensTotal - totalSettled == 0`

**Report fields:** `scenario`, `timestamp`, `scenarioConfig`, `latency` (p50/p95/p99/max), `operationCounts`, `correctnessVerdicts`, `tokenConservation` (including `conserved`).

**Tag filter:** `@Tag("performance")` — runs under `performanceTest`, excluded from default `test` task.

---

### ConcurrentEndByTagIT

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/ConcurrentEndByTagIT.java`

**Scenario:** `tagCount=5` tags, `sessionsPerTag=10` sessions per tag (total 50 sessions), `overlapFactor=0.3` chance each session also carries a second tag. Test uses `@TestMethodOrder` + `@Order` to ensure all streams are running before the concurrent `POST /api/v1/tags/{tag}/end` calls fire.

After all end-by-tag calls complete, the test asserts each tag's `GET /api/v1/tags/{tag}/aggregate` response shows `inFlight.inFlightDebit == 0` — confirming all matched streams settled.

**Conservation check:** Same pattern as ConcurrentSessionsIT — `EndByTagResponse.settledStreams` amounts summed into `totalSettled`.

**Tags:** `@Tag("performance")` + `@Tag("concurrency")` — runs under `performanceTest` (tag expression `performance | concurrency`).

---

### RakeConcurrencyIT

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/RakeConcurrencyIT.java`

**Scenario:** N=10 rake-enabled streams, each with `ratePerSecond=1.0`, `rakeRate=0.1`, `toAccountId="perf-rake-to-{i}"`, `platformAccountId="perf-rake-platform"`. All N streams stop simultaneously via `CountDownLatch` — this is the deadlock-detection test for lock ordering (from→to→platform per D-decision in STATE.md [04-03]).

**Conservation check (closed-pool approach):** Rake is a redistribution. All tokens stay within the FROM + TO + platform account set, so:
- `Σ(finalBalances) == Σ(initialBalances)` — proves no tokens created or destroyed
- `threeWayDelta = initialFrom.subtract(toFinal).subtract(platformFinal).subtract(fromFinal) == 0` — proves from-account debit equals to+platform credits

The `settledAmount` from `StopStreamResponse` is also tracked but not used as the primary conservation signal — it represents the debit from FROM but does not directly map to TO+platform credits in a way that enables a non-redundant assertion.

**Tags:** `@Tag("performance")` + `@Tag("concurrency")`.

## Deviations from Plan

### Deviation: tagPool populated in @Order(1) test method, not in @BeforeAll

The plan's `setupTagsAndStreams()` method is annotated `@Order(1)` and reads `config.getEndByTag().getTagCount()`. Because `PerfTestProperties` requires the Spring context to be active, and `@BeforeAll` runs before the context's `@Autowired` fields are injected (with `@TestInstance(PER_CLASS)` this is fine, but `tagPool` state must survive between the two @Test methods), the setup is performed in `@Order(1)` exactly as the plan specifies — no deviation in intent, but the tagPool fill happens inside the test method body.

### Deviation: ConcurrentEndByTagIT.teardown() does not assertThat(reportFile).exists()

The plan's teardown for Task 2 specifies `PerfReportWriter.write(...)` but the plan action does not explicitly call assertThat on the returned Path (unlike Task 1 which does). Implemented without the Path assertion for consistency with the plan action text — the `conserved` assertion is the load-bearing check. `PerfReportWriter.write()` throws `IOException` on failure so any write error surfaces automatically.

### Auto-merged main before implementation

Wave 0 infrastructure (Plan 01 commits d3858bf, 8dd0555, a5298e1) was not present in the worktree at spawn time. Merged `main` into the worktree branch via fast-forward before implementing the three IT classes. No conflicts.

## Note for Plan 03 (MixedWorkloadIT)

The three patterns established here are the canonical reference for Plans 03-05:

1. **BigDecimal conservation:** Use `AtomicReference.updateAndGet(current -> current.add(amount))` for all concurrent accumulators. Never use a shared `BigDecimal` field with `+=`.
2. **LatencyRecorder:** Instantiate once per class (`private final LatencyRecorder latency = new LatencyRecorder();`), call `latency.record(System.currentTimeMillis() - t0)` per thread, call `latency.snapshot()` once in `@AfterAll`.
3. **PerfReportWriter:** Always in `@AfterAll`. The `write(scenario, reportMap)` call must come before the `assertThat(conserved).isTrue()` assertion so the report is written even if conservation fails (to aid debugging).
4. **Closed-pool conservation for rake:** Use `Σfinals == Σinitials` not `Σsettled == Σdebited`. This is the correct invariant when rake redistributes within the test account pool.
