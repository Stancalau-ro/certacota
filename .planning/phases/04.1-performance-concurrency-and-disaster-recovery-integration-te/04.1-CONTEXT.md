# Phase 04.1: Performance, Concurrency, and Disaster Recovery Integration Test Suite - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

A standalone, config-driven integration test suite that exercises the engine under realistic load: high-concurrency token flows (with organic per-account simulation and hot-account modeling), large-database stress (locally-built seeded Postgres images), and component-failure recovery (Postgres transient outage and Redis restart). All test scenarios run via the public REST API only (no internal bean injection) and produce structured JSON reports with token conservation assertions. Tests are tagged so they do not run in the standard `mvn test` cycle.

No new engine functionality. No changes to production code unless a test reveals a bug. No new REST endpoints beyond what Phase 4 delivered.

</domain>

<decisions>
## Implementation Decisions

### Failure Injection Mechanism
- **D-01:** DB connection loss simulated via Testcontainers `PostgreSQLContainer.pause()` / `unpause()`. No extra containers (Toxiproxy not used). No DataSource mock — real TCP-level failure.
- **D-02:** Two disaster-recovery scenarios:
  1. **Transient outage** — Postgres pause (streams keep running in-memory), then unpause (settlement proceeds). Asserts correct settled amounts and committed balances. Implemented via `DockerClientFactory.lazyClient().pauseContainerCmd(id).exec()` / `unpauseContainerCmd(id).exec()` (NOT Testcontainers `GenericContainer.pause()` — the docker-java API is the stable cross-version path).
  2. **Full Redis restart** — Redis container stop + start via `DockerClientFactory.lazyClient().stopContainerCmd(id).exec()` / `startContainerCmd(id).exec()` (docker-java API), NOT Testcontainers `GenericContainer.stop()` — that API deletes the container and creates a new one with a new port mapping on next start, breaking the JVM's existing Redis port binding (see RESEARCH.md Pitfall 1). The docker-java `stopContainerCmd` + `startContainerCmd` pair preserves the container record and port mapping. While Redis is down, the engine's StreamRegistry loses in-memory state. After restart, the engine reconciles `StreamRegistry` from `streaming_transactions WHERE status = ACTIVE` on the next write (existing Phase 3 reconciliation pattern from STATE.md [04-03]).
- **D-03:** During a Postgres pause, in-flight streams keep running in-memory (StreamRegistry unaffected). Settlement retries succeed when Postgres is restored. Test asserts settled amount = rate × elapsed (not rate × pre-pause time). Engine does NOT auto-terminate streams on DB failure.
- **D-04:** Disaster-recovery tests tagged `@Tag("disaster-recovery")`. Excluded from Maven Surefire plugin. Run explicitly with `mvn verify -Pdisaster-recovery`. Disaster-recovery Maven profile configured in `pom.xml` to activate Failsafe with the `disaster-recovery` tag filter.

### Config-Drive Strategy
- **D-05:** All scenario parameters bound via `@ConfigurationProperties` to `application-perf-test.yml` in the test resources. Activated with `@ActiveProfiles("perf-test")`. Standard Spring Boot mechanism — no custom loaders.
- **D-06:** Concurrency suite parameters: thread count, ramp-up period, steady-state duration, and a `[minSessionDurationMs, maxSessionDurationMs]` interval. Session durations are randomly sampled per account per run using `ThreadLocalRandom`. When min == max, all sessions have identical duration (backward-compatible with fixed-duration scenarios).
- **D-07:** Large-database stress parameters: `stress.accounts` (row count), `stress.transactionsPerAccount` (rows per account), `stress.maxQueryLatencyMs` (latency assertion ceiling).
- **D-08:** Single config file (`application-perf-test.yml`). No environment-specific profiles. CI overrides individual values via `-D` system property flags (e.g., `-Dstress.accounts=500 -Dconcurrency.sessions.rampUpSeconds=2`).

### Concurrency Suite Scope
- **D-09:** Existing `DiscreteTransactionConcurrencyTest` (n=20) and `StreamingConcurrencyTest` (n=10) are left unchanged. They are not promoted, not deleted, and not made config-driven. New tests are added alongside.
- **D-10:** New test classes (all in `engine-service` module, `com.certacota.engine.service` package, tagged `@Tag("performance")` or `@Tag("concurrency")`):
  - `ConcurrentSessionsIT` — N parallel streaming sessions with randomly-varied durations (`[minDurationMs, maxDurationMs]`)
  - `ConcurrentEndByTagIT` — concurrent `POST /api/v1/tags/{tag}/end` calls for overlapping tag sets
  - `MixedConcurrencyIT` — per-account organic simulation with hot-account modeling (see D-11 and D-12)
  - `RakeConcurrencyIT` — N rake-enabled streams settling simultaneously; asserts three-way lock ordering prevents deadlock
- **D-11:** Mixed scenario design (per-account organic simulation): each account in the test pool has an independent event lifecycle running for the configured `durationSeconds`. Events per account: streaming session starts (random count within `[sessions.min, sessions.max]`), discrete credits (random interval within `[creditIntervalMs.min, creditIntervalMs.max]`, amount within `[creditAmount.min, creditAmount.max]`), discrete debits (random interval, amount). No global thread pool assigning accounts — each account's lifecycle is driven by its own goroutine pool. Duration-based (not fixed-op-count).
- **D-12:** Hot accounts: `mixed.hotAccounts.count` accounts randomly selected at test setup. Their operation interval is divided by `mixed.hotAccounts.contentionMultiplier` (e.g., multiplier=3.0 → hot account gets 3× more events than average). Surfaces hot-spot contention patterns (lock-wait starvation, deadlock risk under asymmetric load).
- **D-13:** All new Phase 04.1 tests use the public REST API only — `RestTemplate` or `WebTestClient` against `http://localhost:{port}/api/v1/...`. No `@Autowired` of internal beans (`AccountRepository`, `StreamingTransactionRepository`, etc.). Final balance assertions via `GET /api/v1/accounts/{id}`. Enterprise-portable: tests can be pointed at any deployment of the engine.
- **D-14:** All new concurrency and stress tests capture latency: p50, p95, p99, max across all operations in the run. p95 and p99 are included in the JSON report and optionally asserted against configured thresholds.

### Large-Database Stress — Seeding Strategy
- **D-15:** Data seeded via a locally-built Postgres Docker image. Seed SQL scripts live in `engine-service/src/test/docker/perf-db/`. Image name encodes dataset parameters: `certacota-perf-db-a{accounts}-t{transactionsPerAccount}` (e.g., `certacota-perf-db-a10000-t500000`). Not pushed to any registry.
- **D-16:** At test startup: check if the named image exists locally (`docker images -q {imageName}`). If missing, auto-build using `docker build --build-arg ACCOUNTS={n} --build-arg TXN_PER_ACCOUNT={m} -t {imageName} .` in the seed scripts directory. The image name encoding ensures a dataset size change triggers a rebuild automatically.
- **D-17:** Seed SQL uses Postgres `generate_series()` for bulk INSERT. Seed scripts are parameterized via Docker build args (`ACCOUNTS`, `TXN_PER_ACCOUNT`). Scripts are committed to the repo alongside the Dockerfile.

### Report Format
- **D-18:** Each test class writes a JSON report file: `target/perf-reports/{scenario}-{yyyyMMdd-HHmmss}.json`. Created at test teardown (`@AfterAll`). Directory is `.gitignore`d; CI archives it as a build artifact.
- **D-19:** Report schema (all scenarios):
  ```json
  {
    "scenario": "concurrent-sessions",
    "timestamp": "2026-05-15T10:00:00Z",
    "scenarioConfig": { /* exact @ConfigurationProperties values used */ },
    "latency": { "p50Ms": 12, "p95Ms": 45, "p99Ms": 120, "maxMs": 310 },
    "operationCounts": { "success": 498, "rejected": 2, "error": 0 },
    "correctnessVerdicts": {
      "noDoubleSpend": true,
      "allBalancesCorrect": true,
      "noDeadlockDetected": true
    },
    "tokenConservation": {
      "initialTokensTotal": "1000000.000000000000000000",
      "finalTokensTotal": "924872.341000000000000000",
      "totalSettled": "75127.659000000000000000",
      "totalCredited": "0",
      "totalDebited": "75127.659000000000000000",
      "delta": "0",
      "conserved": true
    }
  }
  ```
- **D-20:** Token conservation is a required assertion in every scenario: `conserved: false` causes the test to fail immediately. Conservation invariant: `Σ(finalBalances) + Σ(allSettlements) + Σ(allDebits) = Σ(initialBalances) + Σ(allCredits)`. Delta of zero = no tokens created or destroyed outside tracked operations. All amounts use `BigDecimal` with 18 decimal places for comparison (consistent with engine arithmetic precision).
- **D-21:** Report diffing: naming convention (`{scenario}-{yyyyMMdd-HHmmss}.json`) allows comparison of any two runs with `jq`. No committed baseline file in v1 of this suite. Manual diff workflow: `jq '.latency.p95Ms' run1.json run2.json`.

<code_context>
## Existing Code Insights

### Reusable Assets
- `TestcontainersConfiguration.java` — static Postgres + Redis containers with `@ServiceConnection` and `@DynamicPropertySource`; extend by exposing container refs as package-visible statics so DR tests can call `.pause()` / `.stop()` / `.start()`
- `DiscreteTransactionConcurrencyTest` / `StreamingConcurrencyTest` — `CountDownLatch` + `ExecutorService.newFixedThreadPool` + `RestTemplate` + `AtomicInteger` (success/fail counters); identical pattern for new tests
- All existing tests use `@SpringBootTest(webEnvironment = RANDOM_PORT)` + `@Import(TestcontainersConfiguration.class)` + `@ActiveProfiles("test")` — same annotations for Phase 04.1 tests plus `"perf-test"` profile

### Established Patterns
- `DefaultResponseErrorHandler` override that returns `false` for `hasError()` — standard in all concurrency tests so non-2xx responses don't throw exceptions
- `RestTemplate` with `HttpMethod.POST` + `HttpEntity<>(body, headers)` — all existing write operations use this; new tests follow the same pattern
- `@BeforeEach` for `RestTemplate` setup — existing pattern; new tests may use `@BeforeAll` for expensive setup (account creation pool)
- `assertThat(finalBalance.compareTo(expectedBalance)).isEqualTo(0)` — BigDecimal equality via `compareTo`, not `.equals()` — required for 18-decimal-place amounts

### Integration Points
- All new tests connect via `http://localhost:{port}/api/v1/...` — same port injection as existing tests
- `target/perf-reports/` directory created by `@BeforeAll` if absent; tests write JSON on `@AfterAll`
- Docker build integration point: `engine-service/src/test/docker/perf-db/` directory with `Dockerfile` + seed SQL scripts + `build-image.sh`

<specifics>
## Specific Ideas

- Token conservation check uses `GET /api/v1/accounts` (list all accounts) or a direct Postgres query via a dedicated `GET /api/v1/admin/token-total` endpoint. If no such aggregate endpoint exists, the test may need to sum balances via individual account GETs or use the audit log endpoint — Claude's discretion on which is cleaner.
- Hot account selection: at `@BeforeAll`, randomly pick `hotAccounts.count` account IDs from the pool. Pass the hot account set to each per-account event runner so it applies the contention multiplier when generating intervals.
- The `scenarioConfig` section of the JSON report should be the exact serialized `@ConfigurationProperties` object used in the run — so the report is self-contained and the exact config that produced the results is always captured alongside them.
- For the Redis restart test: after `redis.stop()`, the test should make a write operation (e.g., attempt to settle a stream) to trigger reconciliation, then verify that `StreamRegistry` correctly reflects the pre-stop state.

</specifics>

<deferred>
## Deferred Ideas

- Committed baseline file for automated p95 regression detection — useful for CI but adds maintenance burden. Revisit after the suite has a few runs to establish realistic baselines.
- Toxiproxy-based latency and packet-loss simulation — more realistic network failure modeling than pause/unpause. Defer to a future perf iteration if transient-fault behavior needs deeper coverage.
- Virtual thread pool configuration for mixed scenario — Java 21 virtual threads could change the contention characteristics significantly. Claude's discretion for now; revisit if thread counts become a bottleneck.
- Throughput measurement (ops/second) as an assertion — currently captured in the report but not asserted against. Add throughput threshold assertions once baselines are established.

</deferred>
