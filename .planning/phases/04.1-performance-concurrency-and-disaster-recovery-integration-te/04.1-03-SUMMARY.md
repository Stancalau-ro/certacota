# Phase 04.1 Plan 03: MixedConcurrencyIT Summary

Per-account organic simulation with hot-account contention modeling. Each of 50 accounts runs an independent event lifecycle (stream starts, credits, debits) on a dedicated platform thread for 30 seconds. Five hot accounts receive events at 3x the normal rate via interval division. Token conservation holds across all concurrent operations.

#### Hot-Account Selection Algorithm

```java
int hotCount = Math.min(config.getMixed().getHotAccounts().getCount(), accountIds.size());
List<String> shuffled = new ArrayList<>(accountIds);
Collections.shuffle(shuffled, new Random(42));
hotAccountIds.addAll(shuffled.subList(0, hotCount));
```

Seeded `Random(42)` ensures the same 5 accounts are designated hot on every test run (default `hotCount=5`). Hot accounts are identified by presence in `hotAccountIds` (a `ConcurrentHashMap.newKeySet()`). The shuffle seed is arbitrary but fixed — Plan 04 can reuse the same seed or choose a different one without affecting correctness.

#### Per-Account Event Distribution

The event loop uses a 0-99 roll to select event type:

| Roll range | Event | Share |
|-----------|-------|-------|
| 0-29 | Stream start (capped at sessionsTarget) | 30% |
| 30-64 | Credit | 35% |
| 65-99 | Debit | 35% |

In practice the stream-start branch fires less than 30% because `sessionsStarted.get() < sessionsTarget` caps it (default target 1-3 sessions per account). Once that cap is reached, event rolls 0-29 fall through — the loop iterates again with the next sleep interval. This means credits and debits account for more than 70% of actual events in steady state, which is the intended organic workload shape.

#### Hot-Account Interval Division

```java
private long sampleInterval(long minMs, long maxMs, boolean isHot, double multiplier) {
    long base = ThreadLocalRandom.current().nextLong(minMs, maxMs + 1);
    if (isHot) {
        return (long) Math.max(1, base / multiplier);
    }
    return base;
}
```

Hot accounts sleep `base / contentionMultiplier` (default `base / 3.0`) between events. With `creditIntervalMs = [200, 1000]`, the hot-account effective range is `[67, 333]ms`. Floor of 1ms prevents a zero-sleep tight loop if `base` rounds down.

#### Token Conservation Invariant (D-20)

```java
BigDecimal delta = initialTokensTotal.get()
    .add(totalCredited.get())
    .subtract(totalFinalBalance)
    .subtract(totalSettled.get())
    .subtract(totalDebited.get());
boolean conserved = delta.compareTo(BigDecimal.ZERO) == 0;
```

All five terms use `BigDecimal.ZERO` comparison (not `equals`) to handle different scales. The `assertThat(conserved).isTrue()` at the end of `@AfterAll` fails the test if conservation breaks — mandatory per D-20.

Concurrent accumulator pattern (canonical per Plan 02 `Note for Plan 03`):
```java
addAtomic(totalCredited, amount);  // = ref.updateAndGet(curr -> curr.add(delta))
```

#### Floor-Rejection Classification Policy

- **HTTP non-201 from credit/debit/stream-start:** `rejectedCount.incrementAndGet()` — expected business-rule rejection (floor violation, idempotency collision, etc.)
- **Exception thrown from `restTemplate.exchange`:** `errorCount.incrementAndGet()` inside `recordCall()`'s catch block — IO/connection failures classified separately
- **Non-200 from stream stop (deadline cleanup):** `rejectedCount.incrementAndGet()` — same policy applied consistently across all write paths

This distinction preserves the ability to differentiate high floor-rejection rates (business logic concern) from high error rates (infrastructure concern) in the JSON report.

#### Teardown: Defensive Stream Stop

`@AfterAll teardown()` iterates `activeStreamIds` (streams not yet stopped by the deadline-stop loop — possible if a runner thread exited via exception before reaching its cleanup section). Each leaked stream is stopped and its `settledAmount` accumulated into `totalSettled` before the conservation check runs.
