# Phase 04.1: Code Review Report

**Reviewed:** 2026-05-15T00:00:00Z
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Reviewed the performance, concurrency, and disaster-recovery integration test suite added in phase 04.1. The suite covers concurrent sessions, rake settlement, end-by-tag concurrency, large-DB stress, and two DR scenarios (Postgres pause, Redis restart).

Four critical defects were found. Two are test-correctness bugs that will cause assertions to fail or silently pass on wrong data: the probe stream token leak in `DisasterRecoveryIT` and the missing debit subtraction in `LargeDbStressIT`. One is a shell arithmetic overflow in `seed.sh` on Alpine/32-bit sh. One is the unguarded Redis container stop with no `finally` block to restart it.

Five warnings cover: hardcoded verdicts in report maps, a NullPointerException path on `settledAmount()`, the timing window in the Postgres recovery assertion, an unchecked docker `images` exit code, and executor resource leaks.

## Critical Issues

### CR-01: Probe stream leaks in-flight tokens — token conservation assertion always fails in DR test

**File:** `engine-service/src/test/java/com/certacota/engine/service/DisasterRecoveryIT.java:205-209`

**Issue:** `redisRestartReconciliation()` starts the probe stream `dr-redis-503-probe` on `dr-redis-acc-1` (line 209) and never stops it. The stream deducts tokens from the account balance before settling. In `teardown()`, `totalFinalBalance` includes the account's reduced balance, but `totalSettled` never receives the probe stream's settled amount. The formula `initialTokensTotal - finalBalance - totalSettled` is therefore negative by the amount that remains locked in the probe stream's in-flight state, causing the conservation assertion on line 302 to always fail.

**Fix:** Stop the probe stream before the stop loop, accumulating its settled amount into `totalSettled`:
```java
// After client.startContainerCmd(redisContainerId).exec() + Thread.sleep(...)
// Stop the probe stream first so its settled amount is counted
ResponseEntity<StopStreamResponse> probeStop = restTemplate.exchange(
    "http://localhost:" + port + "/api/v1/streams/dr-redis-503-probe/stop",
    HttpMethod.POST, new HttpEntity<>(new HttpHeaders()), StopStreamResponse.class);
if (probeStop.getBody() != null && probeStop.getBody().settledAmount() != null) {
    totalSettled.updateAndGet(current -> current.add(probeStop.getBody().settledAmount()));
}
```

### CR-02: Token conservation formula ignores debits — assertion always fails in LargeDbStressIT

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/LargeDbStressIT.java:215-216`

**Issue:** The teardown computes `delta = initialTokensTotal - totalFinalBalance` and asserts it equals zero. However, the test issues up to 50 debit operations (`writesToIssue`, line 145) against `stress_accounts`. Each successful debit reduces account balances, making `totalFinalBalance < initialTokensTotal`. `delta` is positive for every successful debit, so the conservation assertion on line 253 will fail whenever any debits succeed. The `totalDebited` field is tracked but never subtracted.

**Fix:**
```java
// Change line 215:
BigDecimal delta = initialTokensTotal.subtract(totalFinalBalance).subtract(totalDebited.get());
```

### CR-03: Redis container not restarted if any step between stop and start throws — container left permanently stopped

**File:** `engine-service/src/test/java/com/certacota/engine/service/DisasterRecoveryIT.java:199-211`

**Issue:** The Redis stop and restart sequence has no `try/finally` guard:
```java
client.stopContainerCmd(redisContainerId).exec();   // Redis is now down
// ... probe stream call (can throw) ...
client.startContainerCmd(redisContainerId).exec();  // never reached if above throws
Thread.sleep(config.getDr().getRedisRestartWaitMs()); // never reached
```
If the probe stream REST call throws an exception (network timeout, etc.), `startContainerCmd` is never called, the Redis container stays stopped, and subsequent tests or teardown that rely on Redis will hang or fail with opaque errors. The Postgres test (Order=1) correctly uses `try/finally` for unpause; the Redis test (Order=2) does not.

**Fix:**
```java
client.stopContainerCmd(redisContainerId).exec();
try {
    // probe call here
} finally {
    client.startContainerCmd(redisContainerId).exec();
    Thread.sleep(config.getDr().getRedisRestartWaitMs());
}
```

### CR-04: Shell arithmetic overflow in seed.sh on Alpine sh with large row counts

**File:** `engine-service/src/test/docker/perf-db/seed.sh:37`

**Issue:** The echo statement uses `$((${ACCOUNTS} * ${TXN_PER_ACCOUNT}))`. Alpine Linux uses busybox `sh`, whose `$((...))` arithmetic is limited to the platform's C `long` type. On the default Alpine x86_64 build this is 64-bit, but `${ACCOUNTS}=10000` and `${TXN_PER_ACCOUNT}=500000` yields 5,000,000,000 which exceeds a signed 32-bit `long` (2,147,483,647). If the sh on the target Docker host is 32-bit, this silently wraps to 705,032,704, producing a misleading log message. The actual Postgres `generate_series` runs in Postgres and is unaffected, but the echo misleads operators.

**Fix:** Use `printf` or Postgres-side row count for the log, or validate the inputs earlier:
```sh
echo "perf-db seed.sh completed: seeded ${ACCOUNTS} accounts and ${ACCOUNTS}x${TXN_PER_ACCOUNT} audit-log rows"
```
Alternatively, use `awk` for the arithmetic: `awk "BEGIN {print ${ACCOUNTS} * ${TXN_PER_ACCOUNT}}"`.

## Warnings

### WR-01: Hardcoded correctness verdicts in DR report — report always claims success regardless of test outcome

**File:** `engine-service/src/test/java/com/certacota/engine/service/DisasterRecoveryIT.java:293-298`

**Issue:** The `correctnessVerdicts` map in `teardown()` hardcodes `"noFailedSettlements", true` and `"reconciliationVerified", true` regardless of what the test actually measured. The local variables `noFailedSettlements` (line 129) and `reconciliationVerified` (line 215) are scoped to individual test methods and are not fields, so they cannot be referenced from `teardown()`. The report therefore always emits `true` even if settlements failed.

**Fix:** Promote `noFailedSettlements` and `reconciliationVerified` to `AtomicBoolean` fields (following the pattern of `totalSettled`), update them in each scenario method, and reference them in `teardown()`:
```java
private final AtomicBoolean noFailedSettlements = new AtomicBoolean(true);
private final AtomicBoolean reconciliationVerified = new AtomicBoolean(true);

// In teardown:
"noFailedSettlements", noFailedSettlements.get(),
"reconciliationVerified", reconciliationVerified.get()
```

### WR-02: NullPointerException on `settledAmount()` inside `recordCall` lambda in MixedConcurrencyIT

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/MixedConcurrencyIT.java:243-244`

**Issue:** The stop-stream lambda at line 243 calls `addAtomic(totalSettled, resp.getBody().settledAmount())` without a null check on `settledAmount()`. The outer `resp.getBody() != null` guard is present (line 243), but `settledAmount()` itself can return null (e.g., if the response body is a partial JSON with a missing field). A NPE is thrown inside the `recordCall` lambda, which catches it as `Exception e` and increments `errorCount`. This silently corrupts `totalSettled` (the addition never happens) and causes the conservation assertion to fail with a misleading delta.

**Fix:**
```java
if (resp.getStatusCode().value() == 200 && resp.getBody() != null
        && resp.getBody().settledAmount() != null) {
    addAtomic(totalSettled, resp.getBody().settledAmount());
    ...
}
```

### WR-03: Postgres recovery timing window makes the ±10% tolerance structurally unreliable

**File:** `engine-service/src/test/java/com/certacota/engine/service/DisasterRecoveryIT.java:149-164`

**Issue:** The settlement tolerance calculation uses `totalElapsedMs = postMillis - preMillis` where `preMillis` is set before the 500 ms pre-pause sleep and `postMillis` is set after calling three stop-stream HTTP requests. The actual token accrual stops during the `postgresPauseDurationMs` (default 3,000 ms) because writes are blocked. Therefore the expected amount should subtract the pause duration: `expectedPerStream = rate × (totalElapsedMs - pauseDurationMs) / 1000`. With a 3-second pause out of a ~4-second window, `expectedTotal` is inflated by ~75%, and `lowerBound = expectedTotal × 0.90` is far above what actually settled. The assertion will fail whenever the pause duration is a significant fraction of total elapsed time.

**Fix:**
```java
long pauseDurationMs = config.getDr().getPostgresPauseDurationMs();
BigDecimal expectedPerStream = rate
    .multiply(BigDecimal.valueOf(Math.max(0, totalElapsedMs - pauseDurationMs)))
    .divide(BigDecimal.valueOf(1000), 6, RoundingMode.HALF_UP);
```

### WR-04: `docker images -q` exit code is silently ignored — subsequent build proceeds even when docker is unavailable

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/LargeDbStressIT.java:103-108`

**Issue:** `checkProcess.waitFor()` return value is discarded (line 107). If `docker` is not on `PATH` or the daemon is unreachable, the process exits with a non-zero code and `imageId` is empty. The code then falls through to the build step, which also fails — but the resulting `IllegalStateException` at line 124 says "docker build failed", obscuring the real cause (docker unavailable).

**Fix:**
```java
int checkExit = checkProcess.waitFor();
if (checkExit != 0) {
    throw new IllegalStateException(
        "docker images check failed (exit=" + checkExit + "). Is Docker available?");
}
```

### WR-05: ExecutorService not shut down in the `finally` block — threads may leak if await times out

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/ConcurrentSessionsIT.java:154-158`
**Also:** `ConcurrentEndByTagIT.java:189-193`, `RakeConcurrencyIT.java:176-179`

**Issue:** All three tests call `executor.shutdown()` only after the `done.await(...)` assertion passes. If `done.await()` times out (which triggers the assertion failure), `executor.shutdown()` is never called. The thread pool remains alive until GC, holding references to the test class instance and potentially interfering with subsequent tests in the same JVM.

**Fix:** Move executor shutdown into a `try/finally`:
```java
start.countDown();
boolean completed;
try {
    completed = done.await(60, TimeUnit.SECONDS);
} finally {
    executor.shutdown();
    executor.awaitTermination(10, TimeUnit.SECONDS);
}
assertThat(completed).as("All calls completed without deadlock").isTrue();
```

## Info

### IN-01: `performanceTest` and `disasterRecoveryTest` Gradle tasks do not inherit Cucumber system properties

**File:** `engine-service/build.gradle:49-63`

**Issue:** The base `test` task configures `cucumber.junit-platform.naming-strategy` and `cucumber.plugin` system properties. `performanceTest` and `disasterRecoveryTest` omit these. If any Cucumber-based test ever runs under these tasks, output will be un-formatted and the HTML report won't be generated.

**Fix:** Extract shared system properties into a closure and apply it to all three tasks, or configure via `tasks.withType(Test)`.

### IN-02: `PerfReportWriter` uses a relative path — report location is JVM working-directory-dependent

**File:** `engine-service/src/test/java/com/certacota/engine/service/perf/PerfReportWriter.java:22`

**Issue:** `Paths.get("target/perf-reports")` resolves relative to the JVM working directory at the time of the call, which is typically the Gradle project root when run via `./gradlew performanceTest` but undefined when run from an IDE. Reports may be written to an unexpected location.

**Fix:** Resolve relative to a system property set by Gradle:
```java
String base = System.getProperty("certacota.perf.reportDir", "target/perf-reports");
Path reportDir = Paths.get(base);
```
Set `systemProperty 'certacota.perf.reportDir', "${project.buildDir}/perf-reports"` in `build.gradle`.

### IN-03: `DisasterRecoveryIT` does not assert that the probe stream received a 503 response

**File:** `engine-service/src/test/java/com/certacota/engine/service/DisasterRecoveryIT.java:205-209`

**Issue:** The probe stream call is intended to verify that the service returns 503 when Redis is down (per the comment context `dr-redis-503-probe`), but the response code is never checked. The test exercises the code path but does not assert the expected behavior, making it possible for a regression that serves 200 during Redis downtime to go undetected.

**Fix:**
```java
ResponseEntity<String> probeResp = restTemplate.exchange(
    streamsUrl, HttpMethod.POST, new HttpEntity<>(probeBody, probeHeaders), String.class);
assertThat(probeResp.getStatusCode().value())
    .as("Service should return 503 when Redis is unavailable")
    .isEqualTo(503);
```
