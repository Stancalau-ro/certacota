# Phase 04.1 Plan 04: DisasterRecoveryIT Summary

**One-liner:** Postgres pause/unpause + Redis stop/start DR scenarios using DockerClientFactory.lazyClient() with D-13 fallback reconciliation and token conservation assertion.

## What Was Built

`DisasterRecoveryIT.java` implements two disaster-recovery integration test scenarios:

**Scenario 1 ‚Äî Postgres Transient Outage (`postgresTransientOutageRecovery`, @Order(1)):**
- Starts three streaming sessions (dr-pg-stream-1/2/3) at 0.01 tokens/sec
- Pauses the Postgres container via `DockerClientFactory.lazyClient().pauseContainerCmd(containerId).exec()` for `config.getDr().getPostgresPauseDurationMs()` ms (default 3000ms)
- The pause is wrapped in `try/finally` to guarantee `unpauseContainerCmd` runs even if an assertion fails (T-04.1-11 mitigation)
- After unpause + 500ms HikariCP reconnect window, all three streams stop successfully
- Asserts: all stops return 2xx; total settled amount is within ¬±10% of `rate √ó totalElapsedMs / 1000 √ó 3`
- Proves: engine survives transient Postgres outage per D-03 and SC-4

**Scenario 2 ‚Äî Redis Restart with Reconciliation (`redisRestartReconciliation`, @Order(2)):**
- Starts three streaming sessions (dr-redis-stream-1/2/3) at 0.01 tokens/sec
- Stops the Redis container via `DockerClientFactory.lazyClient().stopContainerCmd(containerId).exec()` (NOT `GenericContainer.stop()` ‚Äî that would delete the container and break the JVM's port binding)
- Exercises a 503-probe start attempt while Redis is down (not asserted)
- Restarts Redis via `startContainerCmd(containerId).exec()` and waits `config.getDr().getRedisRestartWaitMs()` ms
- Stops all three streams ‚Äî the engine's D-13 fallback path reads rake fields from Postgres when Redis state is absent
- Asserts: all stops return 2xx; total settled within ¬±20% of expected (wider tolerance for wall-clock fallback)
- Proves: engine reconciles correctly per D-02 + D-13 + SC-4

**@AfterAll teardown:**
- Fetches all six account balances via REST
- Asserts token conservation: `initialTotal - finalBalances - totalSettled == 0`
- Writes `target/perf-reports/disaster-recovery-{timestamp}.json` via `PerfReportWriter`

## Key Design Decisions

1. **`@Import(DrContainerHolder.class)` not `TestcontainersConfiguration.class`** ‚Äî `TestcontainersConfiguration` exposes Postgres as a Spring-managed `@Bean` with no accessible container ID. `DrContainerHolder` uses `static final` fields accessible via `getPostgresContainer().getContainerId()` and `getRedisContainer().getContainerId()`, required by `DockerClientFactory.lazyClient()` pause/stop commands.

2. **try/finally around pauseContainerCmd** ‚Äî The plan listed this as "executor discretion" per T-04.1-11. Applied because a test assertion failure between pause and unpause would leave Postgres paused for subsequent runs. The finally block guarantees `unpauseContainerCmd` runs.

3. **Wave 0 dependency files included in this worktree** ‚Äî This worktree was branched before Waves 0-2 merged to main. Files are byte-identical to the main branch versions from `agent-a875a8089334528c1`. This is a Rule 3 deviation (blocking issue auto-fix): without these files the compilation fails.
