**Per-account organic simulation** (user-initiated topic): User asked whether scenarios would mix discrete and streaming transactions realistically with credits/debits interleaved. Outcome: each account in the pool has an independent event lifecycle. Per-account events: streaming sessions, credits, debits — all at random intervals within configured [min, max] ranges. Duration-based, not op-count-based.

**Hot accounts** (user-initiated topic): User specified that a subset of accounts should suffer x% more contention than average. Config: `mixed.hotAccounts.count` and `mixed.hotAccounts.contentionMultiplier`. Hot accounts receive `multiplier`x more events (interval / multiplier).

---

**Report contents:** User asked about a checksum on the total body of tokens to ensure no tokens appeared or disappeared in an untrackable way. Outcome: token conservation check added as a required assertion (`conserved: false` = test failure). Conservation invariant: `Σ(finalBalances) + Σ(allSettlements) + Σ(allDebits) = Σ(initialBalances) + Σ(allCredits)`.

---

| Option | Description | Selected |

**User's choice:** Naming convention only ({scenario}-{yyyyMMdd-HHmmss}.json). No committed baseline in v1.
